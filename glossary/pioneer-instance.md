# pioneer instance

- **Class**: DERIVED (meta concept naming the relationship between framework and originating deployment; not a primitive with framework-level instances of its own)
- **Layer**: framework-meta (operates on framework + deployments without being one)
- **Axis**: cross-axis (pioneer instance instantiates all axes; isn't itself axis-bound)
- **VISION usage**: not directly named in current VISION (post Phase 1.85 clean-stance restructure relocated pioneer-specific framing to `PIONEER.md`); implicit via VISION's falsification framing — falsification depends on deployment evidence, and pioneer instance provides that evidence base while framework matures

**Canonical**: The originating deployment of a framework — typically the first / seminal implementation — which serves three purposes simultaneously:

1. **Production-tool**: the practitioner who pioneered the deployment uses it for actual professional work (not demo, not testing)
2. **Research-lab**: real work surfaces requirements that drive framework evolution; deployment experience is upstream signal for framework refinement
3. **IP-proving-ground**: the deployment's outcomes (successes / failures / regulatory challenges) validate or falsify framework claims; pioneer evidence is load-bearing while alternative-deployment evidence is sparse

For PBS specifically: PBS-Schulz is the pioneer instance — Gunther Schulz's planning bureau workspace at his Planungsbüro. The framework was extracted FROM that work, then re-applied TO it; refinements flow bidirectionally between framework and pioneer. See `PIONEER.md` for full pioneer-instance identity-anchor.

**What it is**: A meta-concept naming the relationship between framework and originating deployment. Not specific to AI frameworks — applies to any methodology / framework / tool that emerges from real practice. The triple-purpose framing distinguishes pioneer instance from generic deployments: most deployments are JUST production tools; pioneer instances also produce framework evolution feedback + accumulate IP-validating evidence.

The pioneer instance bears load-bearing weight for the framework's empirical validation — claims unsupported by pioneer evidence are speculative until additional deployments accumulate. As alternative deployments multiply, evidence base broadens; pioneer status gradually shifts from primary validator to historical anchor.

**What it is NOT**:
- Not a primitive type — pioneer instance is a ROLE / RELATIONSHIP a workspace takes on, not a kind of entity
- Not specific to AI frameworks — applies to any extractable methodology
- Not implying first-deployment-is-best — pioneers evolve based on later deployments' feedback (pioneer is starting point, not endpoint)
- Not permanent — once second/third deployments accumulate evidence, pioneer instance's role shifts; pioneer status becomes historical context rather than active validation source
- Not synonymous with `workspace` — workspace is the architectural primitive; pioneer instance is the ROLE a workspace plays when serving the originating-deployment function
- Not "the canonical workspace" — there is no canonical workspace; pioneer is just the first/originating one

**Cross-archetype illustration**:

*Pioneer instance manifestations per archetype*:
- **PBS-Schulz** (pioneer of PBS framework): planning bureau deployment; production tool for B-Plan-Begründung work; research lab for framework evolution; proving ground for axis claims (intertwining viability, sparring effectiveness, defensibility under EU regulatory pressure)
- *Hypothetical parallels*: a research lab that pioneers a research-methodology framework; a legal practice that pioneers a legal-AI framework; a creative studio that pioneers a creative-collaboration framework — all would serve same triple-purpose role for their respective frameworks

In all cases: the pioneer instance is where framework theory meets practice; bidirectional refinement happens here; framework claims survive or fail based on pioneer evidence (until evidence base broadens).

**Boundary test**: Three questions — pioneer instance role applies when ALL resolve favorably:
1. Is this deployment serving as a **production-tool** (practitioner uses it for actual professional work, not demo / testing)?
2. Is this deployment serving as a **research-lab** (real work surfaces requirements that drive framework evolution)?
3. Is this deployment serving as an **IP-proving-ground** (deployment's outcomes validate or falsify framework claims)?

If 1+2+3 yes → pioneer instance role applies. If only 1 (production-tool) → regular deployment, not pioneer.

Negative-marker test (deployment is NOT pioneer instance):
- Deployment uses framework but doesn't surface framework-evolution requirements
- Deployment outcomes don't feed back into framework refinement
- Deployment is one of many; evidence base already broadened beyond it
- Framework was developed independently of this deployment

**Cardinality + lifecycle**:

**Cardinality**: typically 1 per framework (the originating deployment). Could be MULTIPLE pioneers if framework emerges from parallel originating efforts where multiple deployments contributed jointly to framework extraction. PBS has 1 pioneer (PBS-Schulz). Multi-pioneer cases are theoretically possible but less common; would require parallel evidence-aggregation discipline.

**Lifecycle**:
- **Begins**: at framework extraction — when the deployment's patterns crystallize into framework primitives
- **Active phase**: while framework relies primarily on pioneer evidence (alternative deployments sparse or absent)
- **Transition**: as additional deployments accumulate, pioneer status gradually shifts from primary validator to historical anchor; framework's evidence base broadens beyond pioneer's data
- **Mature phase**: pioneer status becomes historical context; framework claims now have multi-deployment evidence base; pioneer no longer load-bearing for individual claims, but remains origin reference

For PBS: currently in active phase — PBS-Schulz is primary evidence base; second/third deployments not yet accumulated.
