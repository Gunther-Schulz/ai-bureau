## B. Triad applied per path

### B.1 — Inbound tool-call failure (NEW shape per D74; HYBRID — reuses D73 §B.3 inbound-handler precedent translated to MCP McpError surface)

| Triad element | Lock |
|---|---|
| **Detection** | The default factory's `@server.call_tool()`-registered handler delegates to `adapter.route_tool_call(tool_name, arguments)`. Detection points: (1) workspace not attached → `McpError(ErrorData(code=-32603, message="adapter not attached..."))`; (2) tool name has no `.` separator → `McpError(ErrorData(code=-32601, message="expected format '<binding_id>.<skill_id>'"))`; (3) binding_id has no matching specialist in `substrate.specialist_instances` → `McpError(ErrorData(code=-32601, message="no specialist binding <bid>"))`; (4) skill_id not present on the specialist OR present but `publicly-exposed != True` → `McpError(ErrorData(code=-32601, message="skill <sid> not exposed by specialist <bid>"))`; (5) `specialist.handle_skill(skill_id, params)` raises (D50 SkillExecutionError, AdapterCallError from chained adapter.call, or arbitrary Exception) → `McpError(ErrorData(code=-32603, message="skill <sid> on specialist <bid> failed: <type>: <msg>"))`; (6) specialist directly raises `McpError` → propagated WITHOUT re-wrap. Per D48 §D D-1: no PRE-WIRE action event emitted at the adapter side (inbound path; the specialist's handle_skill emits its own attribution-bound action event per D50 + D64 — the framework's chain integrity is preserved by the specialist's own emission, not by the inbound adapter). |
| **Surface** | `McpError` raised from the @call_tool handler is caught by the MCP SDK's `Server` request-dispatch loop and converted to a `CallToolResult` with `isError=True` + a `TextContent` block carrying the McpError's `ErrorData.message`. The ClientSession's `call_tool` call delivers this `CallToolResult(isError=True)` to the caller — the caller inspects `isError` + `content[0].text`. NOT `AdapterCallError` — this adapter IS the server; the framework process is the receiver of the inbound call, not the caller. The McpError + JSON-RPC error code IS the surface (external MCP client would see the same shape; in tests, the in-process ClientSession's `call_tool` return value carries it). Server task stays alive — per-call exceptions do not abort the server. JSON-RPC error code -32601 (Method not found) covers unknown-tool + not-exposed cases; -32603 (Internal error) covers handle_skill exception cases. Per JSON-RPC 2.0 §5.1 + MCP spec error-code conventions. Original exception chained via Python's `__cause__` (`raise ... from exc`) when the handler wraps non-McpError exceptions. |
| **Recovery** | Server stays running (the MCP SDK Server task continues serving subsequent requests). The MCP client (test harness OR external client OR specialist composed into a downstream chain via D71 `RealWireMCPClientAdapter` cross-call) observes the `CallToolResult(isError=True)` + decides per-error-code recovery (retry / abort / surface to operator). Framework does NOT mandate retry semantics — the JSON-RPC error code is the caller-policy hint. Chain integrity preserved per D10 + D39 + D40 §A: the specialist's `handle_skill` body may have emitted an action event PRE-failure (per D5 §I3 accountability anchor + D50 §B.1 surface contract); if it did not (failure before emit), the event chain remains consistent because the inbound adapter itself does NOT pre-emit (unlike outbound adapters D71/D72/D73 which pre-emit per D48 §D D-5 resolution at D71 §B.1). |

## D. What is NOT in this decision

The five explicit DEFERS surfaced by combining D71 (MCP SDK reuse) + D73 (server-side aggregator) deferral patterns + c6-specific NEW-surface concerns:
