# D48 — 2026-05-12 — Extends D16 — adapter cluster honors detection-surface-recovery triad (per D45 §C)

**Decision (substantive; third cluster supersedes per D45 §C)**: The adapter runtime path — `call()` invocation failure semantics + `attach_workspace` post-boot failure semantics + the specialist-side required-adapter-binding resolution failure surface — is locked under D45's detection-surface-recovery triad. ## A. Scope of cluster

This entry applies the D45 triad to three related adapter-domain failure paths. **Scope is request/response invocation per D16's pattern (1)** — the `call()`-centric framing fits request/response (MCPToolAdapter + DirectAPIAdapter shipped in Phase B both fit pattern 1); D16's two other named patterns (delegation peer + passive event source) have different failure-mode shapes (delegation handshake; emit-on-watch / source-disconnect / source-error) — deferred to those patterns' first impls per §D D-4. `impl/src/fresh_plan/runtime/adapter.py:10-12` docstring already flags this scoping; D48 codifies the framework-level contract.

The three paths unified here:

- **A.1 Adapter `call()` failure shape (Phase C real-wire contract)**: Phase B stubs (`adapter.py:135-150` MCPToolAdapter + `adapter.py:165-181` DirectAPIAdapter) emit one action event via `_emit_action` then return a canned `{ok: True, stub: True, ...}` dict — cannot fail meaningfully at Phase B level. D48 locks the **forward-bar contract**: real-wire impls (subclasses of `Adapter` overriding `call()`) SHALL raise `AdapterCallError` (NEW typed exception) on protocol / transport / auth / timeout / upstream failures. Surfaces both directly (caller of `adapter.call(...)` catches) and indirectly (when invoked from inside a specialist's `on_event` → substrate's `_dispatch_event_to_subscribers` per D37 + D44 → captured per D47 §B.1 into `_subscriber_failures` → aggregated as `SubscriberDispatchError` after outer drain).

- **A.2 Adapter `attach_workspace` failure (Phase C real-wire contract)**: `adapter.py:72-75` is trivial assignment in Phase B (`self._workspace = workspace; self._emit_event = workspace._emit_event`) — cannot fail. `boot.py:260-261` invokes `for adapter in substrate.adapter_instances.values(): adapter.attach_workspace(workspace)` without try/except. For Phase C real-wire, attach may run connection pool setup / auth handshake / pre-flight / config validation; D48 locks the boot-site wrapping contract so attach-time failures surface as structured `WorkspaceBootError` rather than uncaught propagation.

- **A.3 Specialist required-adapter-binding resolution failure**: `specialist.py:103-106` raises bare `RuntimeError(f"specialist {self.id!r}: required-adapter-binding {required!r} has no matching adapter-binding in workspace")` inside `attach_workspace`. Called from `boot.py:267-270` without try/except wrapper. D48 replaces with structured `WorkspaceBootError(failures=[ValidationFailure(category="adapter-binding-resolution", ...)])` per D46 pattern. Adjacent cleanup — the audit's "root cause lost, mis-attributed downstream" framing (D45 §C item 3) originally surfaced this case; D46 closed the worse upstream version (unknown adapter provision-id at `boot.py:189-214`), leaving this consumer-side case for D48.

Out of scope (different clusters):

- Adapter binding mid-boot via unknown adapter provision-id (closed by D46 §B.1).
- Specialist `handle_skill` failure shape + `on_event` exception path (D49 specialist cluster — though D47 §B.1 already covers the on_event capture mechanism at substrate level; D49 will spec `handle_skill`).
- D30 §4 per-work-unit identity checks; B1 collect-all skipping (D50 validation cluster).
- Composition-change post-projection state validity around adapter bindings (D51 composition-change cluster).
- Delegation peer + passive event source pattern failure modes (D16 patterns 2 + 3); deferred to those patterns' first impls (§D D-4).

## B. Triad applied per path

### B.1 — Adapter `call()` failure (Phase C real-wire forward-bar)

| Triad element | Lock |
|---|---|
| **Detection** | Real-wire adapter impls (subclasses of `Adapter` overriding `call()`) SHALL raise `AdapterCallError` (NEW typed exception) on protocol-level failures. Starting category vocabulary: `transport` / `auth` / `timeout` / `protocol-error` / `upstream-error` / `unknown`. Per-protocol extensions MAY register additional categories per D29 namespacing (e.g., gRPC `Status` codes, websocket close codes, message-bus delivery semantics — see §D D-3). |
| **Surface** | `AdapterCallError` carries structured fields: `adapter_id` (the adapter spec's id), `call_target` (the `tool_name` parameter), `category` (string; starter vocabulary above; extension-registrable), `detail` (dict carrying protocol-specific structured context), original exception chained via Python `from`. Caller gets all-fields-visible diagnostic without reading server logs. When invoked indirectly from inside a specialist's `on_event` (subscriber-dispatch path per D37 + D44 + D47), the exception is captured per D47 §B.1 into substrate's `_subscriber_failures` (`substrate.py:310-320`) and aggregated as `SubscriberDispatchError(failures)` (raised at `substrate.py:274-286` after outer drain) — caller of the outermost `append_event` sees the wrapper; per-specialist failure detail accessible via `SubscriberDispatchError.failures[i][2]` (the captured exception). |
| **Recovery** | Caller catches `AdapterCallError`. Per-category recovery strategies (retry / fail-fast / circuit-breaker / fall-through) are caller-policy concern — Phase C+ shape policy or specialist policy decides. Framework does NOT mandate retry semantics. Chain integrity is preserved per D10 + D39 + D40 §A regardless of `call()` failure: if action event was emitted before the failure (Phase B stub convention; Phase C choice deferred per §D D-5), the event stands in the chain with no outcome (caller may emit a subsequent `state-change` event to record outcome); if no event was emitted, chain is unchanged. |

### B.2 — Adapter `attach_workspace` failure (Phase C real-wire forward-bar)

| Triad element | Lock |
|---|---|
| **Detection** | Real-wire adapter impls SHALL raise an exception (typed at the adapter-impl's discretion; framework does NOT prescribe vocabulary for attach-time-internal exceptions because attach-failure causes are protocol-specific per §D D-2) when attach pre-flight fails. The substrate-side boot wrapping converts to the framework's failure surface. |
| **Surface** | `boot.py:260-261` adapter-attach loop wraps each `adapter.attach_workspace(workspace)` call in try/except → re-raise as `WorkspaceBootError(failures=[ValidationFailure(category="adapter-attach", path=f"composition.adapter-bindings[binding-id={bid!r}]", value=adapter.id, reason=f"adapter {adapter.id!r} attach failed: {underlying-exception}")])`. Caller sees which adapter binding failed + the underlying cause via Python's exception chaining. |
| **Recovery** | Boot is aborted per D46 §B.2 partial-and-discardable semantics: substrate object exists with partially-bound adapters (0..idx-1 attached; idx..N not attached); `Workspace` handle is NOT returned (the `WorkspaceBootError` raises before `return workspace` at `boot.py:365`). For Phase B stubs no failure mode exists; the wrapping is defensive for Phase C+. Phase C+ real-wire substrates SHALL define teardown semantics on partial-attach failure (e.g., closing connection pools of already-attached adapters). |

### B.3 — Specialist required-adapter-binding resolution failure

| Triad element | Lock |
|---|---|
| **Detection** | Currently `specialist.py:103-106` raises bare `RuntimeError`. D48 replaces with `WorkspaceBootError(failures=[ValidationFailure(...)])` raised at the specialist site (symmetric with D46's raise-at-failure-site pattern — keeps the failure-emitting code self-contained; boot.py's outer wrapping is symmetry-only since the structured form already raises from the specialist). |
| **Surface** | `WorkspaceBootError(failures=[ValidationFailure(category="adapter-binding-resolution", path=f"composition.specialist-bindings[binding-id={bid!r}].required-adapter-bindings[{idx}]", value=required_provision_id, reason=f"specialist {self.id!r}: required-adapter-binding {required!r} has no matching adapter-binding in workspace")]). Caller sees which specialist + which required-adapter-binding entry failed + the unresolved provision-id. |
| **Recovery** | Boot is aborted per D46 §B.2 partial-and-discardable semantics; substrate object partial; no `Workspace` handle returned. Caller fixes the manifest (correct the `required-adapter-bindings[<idx>]` provision-id to match a bound adapter) and re-boots. |

## D. What is NOT in this decision

- **No change to D16 adapter contract slots** — D16's slot definitions stand. D48 extends D16's runtime semantics around failure handling without altering the kind contract. `adapter.schema.json:16` has `additionalProperties: false`; no new slots can be added without schema amendment, which D48 explicitly does NOT do.

- **No retry / backoff / circuit-breaker / rate-limit / quota / auth-token-refresh / fall-through semantics** — caller policy (Phase C+ shape or specialist concern). Framework names the failure-detection + diagnostic-surface + abort-cleanly contract; recovery-as-retry is policy.

- **No connection-pool / connection-lifecycle / teardown specification for real-wire adapters** — Phase C+ impl concern. D48 specifies the framework-level attach-failure surface; per-protocol attach mechanics are extension-defined.

- **No specific real-wire protocol error-code mapping** — per-protocol extension territory. Each protocol extension (`mcp-server-ext` for MCP, `direct-api-ext` for direct-api, future Phase C protocol extensions for HTTP / gRPC / websocket / etc.) maps its native error codes to the `AdapterCallError.category` + `detail` shape per its own conventions.

- **No async adapter `call()` interface** — Phase C+ async substrates territory.

- **No retroactive rewrite of D16, D29, D37, D44, D46, D47 entries** — append-only ledger. D48 EXTENDS those entries; their original wording stands.

The five **explicit DEFERS** surfaced by the pre-lock probe (§E):

- **D-1 — `AdapterCallError` call-lifecycle raise-point deferred to Phase C**: WHEN in `call()` the exception raises — before-wire (parameter validation), mid-wire (network / protocol), or after-wire (response parsing) — is per-real-wire-impl choice. D48 specifies the typed exception + structured fields; the raise-point is operational detail Phase C real-wire impls fix per their conventions.

- **D-2 — Adapter `attach_workspace` failure cause-vocabulary deferred to Phase C+ per-protocol-extension**: WHAT kinds of attach failures can occur (connection-pool / DNS / TLS-cert / auth-handshake / config-validation / etc.) is protocol-specific. D48 names the `adapter-attach` category at the framework surface; the cause-vocabulary inside each adapter-impl's own raised exception is extension-defined and registered per D29 if cross-extension visibility is needed.

- **D-3 — `AdapterCallError` category vocabulary extension for non-HTTP transports deferred to per-protocol extensions**: the starter vocabulary (`transport` / `auth` / `timeout` / `protocol-error` / `upstream-error` / `unknown`) maps cleanly to HTTP / JSON-RPC. Non-HTTP transports (gRPC `Status` codes, websocket close codes, message-bus delivery semantics, file-sync stat errors) have different natural categories. Per-protocol extensions MAY register additional categories per D29 namespacing; D48 does not lock the full vocabulary.

- **D-4 — Delegation peer + passive event source pattern failure modes deferred to those patterns' first impls**: D16's patterns 2 (delegation peer) + 3 (passive event source) have failure-mode shapes distinct from request/response: delegation involves handshake protocols; passive sources involve emit-on-watch / source-disconnect / source-error. D48 scope is request/response (D16 pattern 1) because Phase B's MCPToolAdapter + DirectAPIAdapter both fit pattern 1; pattern-2/3 failure-mode locks land when first impls of those patterns arrive (Phase C+).

- **D-5 — Action-event-timing relative to `call()` deferred to Phase C real-wire**: Phase B stub convention is emit-action-event-BEFORE-returning (per `adapter.py:143` + `adapter.py:173`); the event represents intent-recorded. Phase C real-wire impls choose between: (a) emit-before-call (action = intent-recorded; failure → action event in chain with no outcome; subsequent `state-change` event by caller can record outcome); (b) emit-after-success (action = outcome; failure → no event + `AdapterCallError`); (c) emit-pending-then-state-change-after-completion (two-event pattern; finest-grain audit trail). D48 does NOT lock; future D-entry locks per Phase C real-wire learning. The choice has audit-trail implications (D5 §I3 attribution + D10 chain integrity around partial-outcome events).

## Rationale

The 2026-05-12 failure-mode audit identified adapter `call()` failure-shape as one of D45's named SUSPECTs but with framing concentrated on the call-time path. D46 closed the boot-time-resolution path (`load_adapter_from_provision` unknown-provision-id surfaces as structured `WorkspaceBootError`); D48 closes the remaining adapter-domain paths.

The cluster is intentionally THIRD in the D45 §C sequence (boot → subscriber-dispatch → adapter): adapter integration is the natural extension of the substrate's append-event flow once subscriber-dispatch is locked. Adapter-emitted events traverse the full 9-step `append_event` sequence (D44 + D47); adapter-call failures inside specialist `on_event` handlers compose with D47's subscriber-exception aggregation. Locking the adapter triad here sets the contract that Phase C real-wire impls (Claude Agent SDK substrate replacing the in-process stub; real MCP server adapter replacing the stub; real direct-API adapter replacing the stub; etc.) have a clear bar to meet, not a happy-path-only example to reverse-engineer from.

The §B.3 bare-RuntimeError cleanup at `specialist.py:103-106` is adjacent to the adapter cluster framing — the failure is about adapter-binding-resolution from the specialist's consumer side, complementing D46's adapter-binding-resolution from the boot resolver side. Folding it here keeps adapter-binding-resolution failure-mode contracts in one place rather than splitting between D48 + D49.

The 5 explicit DEFERS (§D D-1 through D-5) honor the durability-vs-scaffolding framing (CONCEPTS "What is durable vs scaffolding"): D48 is *specification* — the typed exception + category surface + structured field shape — not Phase C real-wire *implementation*. Locking what's specification today + naming what's Phase C real-wire choice keeps the framework durable across the eventual real-wire integrations. The pre-lock probe surfaced the deferrals as a class — without the probe, they would have been quietly imported (e.g., the action-event-timing convention would have inherited Phase B's emit-before-return as an unnamed assumption; the HTTP-shaped category vocabulary would have implicitly excluded non-HTTP transports).

D48 follows D46 + D47 structural template with one addition: §E now documents the probe disposition (FIRED, not SKIPPED) since this is the first cluster supersedes to introduce new contract content beyond pure audit cleanup. Future audit-driven entries with new contract content cite this §E as precedent.
