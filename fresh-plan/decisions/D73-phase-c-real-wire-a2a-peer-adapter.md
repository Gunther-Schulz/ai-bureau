Caller fixes the manifest (changes `configuration.port` to an unoccupied value OR uses 0 for auto-allocate) OR fixes the deployment environment (frees the occupied port) and re-boots. |

### B.2 — Outbound `call()` real-wire failure (parallel to D72 §B.1)

| Triad element | Lock |
|---|---|
| **Detection** | `A2APeerAdapter.call()` synchronously drives `httpx.AsyncClient.post("/tasks", json=task_payload)` via `asyncio.run`. Detection points are IDENTICAL to D72 §B.1: (a) httpx module import failure → `transport`; (b) `_session_factory` / `AsyncClient` construction raising → propagated up; (c) `httpx.ConnectError` / `httpx.NetworkError` → `transport`; (d) `httpx.TimeoutException` (subclasses included) → `timeout`; (e) `httpx.HTTPStatusError` with status_code in {401, 403, 407} → `auth` (NATIVE HTTP mapping per RFC 7235 + 7239 per D72 §B.1); (f) other 4xx / 5xx → `upstream-error`; (g) `httpx.RemoteProtocolError` / `httpx.DecodingError` → `protocol-error`; (h) `response.json()` `ValueError` (json.JSONDecodeError subclass) → `protocol-error`; (i) `OSError` → `transport`; (j) catch-all `Exception` → `unknown`. Per D48 §D D-1: action event emitted PRE-WIRE; AdapterCallError can raise from anywhere downstream. |
| **Surface** | `AdapterCallError` raised carrying structured fields: `adapter_id` (the adapter spec's id; `a2a-peer-adapter` for the 0.1.0 provision); `call_target` (the `tool_name` parameter — interpreted as the peer's skill_id in the JSON task payload's `skill_id` field; URL path is the constant `/tasks`); `category` (string; starter vocabulary per D48 §B.1); `detail` (dict with HTTP-specific structured context). Original exception chained via Python's `__cause__` (`raise ... from exc`). Composition with D47 §B.1 SubscriberDispatchError aggregation preserved when call fires from inside a specialist's `on_event`. |
| **Recovery** | Caller catches `AdapterCallError`, inspects `.category`, decides per-category recovery (retry / fail-fast / circuit-breaker / credential-refresh-on-auth). Framework does NOT mandate retry semantics per D48 §B.1 caller-policy framing. Chain integrity preserved per D10 + D39 + D40 §A: the `action` event emitted PRE-WIRE stands in the chain regardless of `call()` outcome (per D48 §D D-5 resolution at D71 §B.1 + D72 §B.1 carryforward). |

### B.3 — Inbound handler failure (specialist.handle_skill raises during POST /tasks routing) — NEW SURFACE

| Triad element | Lock |
|---|---|
| **Detection** | `_make_a2a_request_handler`-built handler's `do_POST` invokes `adapter.handle_inbound_task(payload)` which calls `specialist.handle_skill(skill_id, params)`. Three distinct detection points: (1) JSON decode failure on inbound body → `ValueError` caught in `do_POST`; (2) skill_id not found OR not publicly-exposed → `KeyError` raised in `handle_inbound_task`; (3) `specialist.handle_skill` raises any exception (SkillExecutionError per D50 §B.1, AdapterCallError from chained adapter.call per D48 §B.1, or arbitrary Exception). |
| **Surface** | Handler catches each detection class and writes an A2A-shaped JSON error response with the appropriate HTTP status: (1) malformed JSON → HTTP 400 + `{"error": {"code": "malformed-json", "message": str(exc)}}`; (2) skill not found → HTTP 404 + `{"error": {"code": "skill-not-found", "message": str(exc)}}`; (3) handle_skill exception → HTTP 500 + `{"error": {"code": "skill-execution-error", "type": type(exc).__name__, "message": str(exc)}}`. NOT AdapterCallError — server-side; the framework process is the receiver, not the caller. The HTTP error response IS the surface (external A2A peer would see it; in tests, the in-process httpx.Client observes it). |
| **Recovery** | Server thread continues serving (per-request error does not abort the server). External A2A peer (or test harness) sees the HTTP error response + decides per-category recovery (retry / abort / surface to upstream operator). The handler swallows the exception locally so the server.serve_forever loop is not interrupted; the underlying specialist.handle_skill exception is preserved in the response body's `message` field for diagnostic visibility. |

The choice to **combine publisher + task-receiver + outbound peer-client in ONE adapter class** (rather than three separate adapters) honors D21 §"What this requires from extensions" naming "A2A peer adapter" as a single composition slot — aggregates skills + serves agent-card + routes incoming tasks. Trade-off: the class is larger (~434 LOC including docstrings) but cohesion is high — all three concerns share the workspace handle + the same server thread.

The choice to **use stdlib `http.server.ThreadingHTTPServer` on a background `threading.Thread(daemon=True)`** (rather than aiohttp / asyncio HTTP server) honors D44 "no async substrate model" + coherence-advisory #3 in the c5 tracker. It does NOT introduce an async substrate model at the framework level (the substrate remains sync per D44). When async-substrate work lands (Phase C+ refinement per D48 §D D-4), the server thread can convert without architectural rework.

The choice to **use `a2a-sdk` (canonical types via protobuf) rather than from-scratch dataclasses** honors D68 §B.3 "use canonical SDK when API fits." Trade-off: 12 transitive deps (protobuf + google-api-core + google-auth + requests + json-rpc + googleapis-common-protos + protobuf supporting libs) vs ~50 LOC of from-scratch dataclasses.
