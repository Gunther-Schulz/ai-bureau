## B. Triad applied per path

### B.1 — SDK init / auth / connection failure (Phase C real-wire forward-bar — NEW)

| Triad element | Lock |
|---|---|
| **Detection** | Any exception raised by the SDK during `start_session` is caught: (a) `ImportError` if `claude_agent_sdk` module not installed at the time `start_session` resolves the default client factory; (b) factory-construction exceptions (e.g., SDK constructor rejecting `ClaudeAgentOptions`); (c) `client.connect(prompt)` exceptions surfaced via `asyncio.run` (`CLIConnectionError` / `CLINotFoundError` / `ProcessError` / `ClaudeSDKError` / any other underlying transport exception); (d) exceptions from `client.receive_response()` iteration. Framework-level exceptions raised from inside `append_event` during translation (`WorkspaceBootError`, `MalformedEventError`, `EventRejected`, `SubscriberDispatchError`, `HookExecutionError`) are caught separately and surface UNCHANGED — the SDK-side path completed; the framework rejected the resulting event, which is a different failure class. Double-`start_session` invocation while a session is already active is detected via the `_session_active` flag check at method entry. |
| **Surface** | `WorkspaceBootError` raised carrying `[ValidationFailure(category="sdk-init", path="substrate.session.start", value=<sdk-class-name>, reason=<diagnostic>)]`. The diagnostic string is `"SDK session failed: <underlying-exception>"` for runtime failures, `"SDK client construction failed: <underlying-exception>"` for factory failures, `"claude-agent-sdk import failed: <ImportError>"` for missing-module failures, `"start_session called while session already active"` for double-invocation. Original SDK exception chained via Python's `__cause__` attribute (`raise ... from exc`) — caller sees full traceback chain. `WorkspaceBootError` reused (not a new typed exception) because `sdk-init` failures share `WorkspaceBootError`'s abort-and-discard semantics with boot-time failures: the session never fully started; no observable state survives; caller's response (retry / abort / surface) is identical in shape. Per D48 §B.1 + D50 §B.1 precedent for category-vocabulary extension over new-typed-exception when the recovery contract aligns. |
| **Recovery** | Caller catches `WorkspaceBootError`, inspects `.failures[0].category == "sdk-init"`, decides retry vs abort. Substrate is left in a clean state by `start_session` (or `stop_session` if disconnect raised): `_sdk_client` is None and `_session_active` is False, so a subsequent `start_session` call is valid. Chain integrity preserved per D10 + D39 + D40 §A — no half-emitted event lands in the chain when the SDK round-trip failed before translation (the failure raises before `append_event` is called for that message). Per-category recovery strategies (retry-with-backoff for transport errors; refresh-and-retry for auth errors; user-prompt for credential-config errors) are caller-policy concern per D48 §B.1 framework-does-not-mandate-retry pattern. |

## Rationale

D68 §A locked C1 as the first Phase C workstream + foundational (per dependency table — C2/C3-C6 depend on C1). D68 §B.1 locked the Claude Agent SDK as the substrate library; §B.3 named the verify-at-workstream-start discipline as the SDK-claim-grounding mechanism. D69 instantiates both — installs the SDK, verifies the API, lands the substrate impl, surfaces the first Phase C real-wire failure path (`sdk-init`) per D45's standing triad requirement.

The choice to **add a new substrate alongside InProcessSubstrate + MSAgentFrameworkSubstrate** (rather than replace InProcessSubstrate) honors D41 two-substrate parity + the parallel growth pattern set by Phase B. Each substrate impl is its own extension at `extensions/<id>-substrate-ext/0.1.0/`; the registry dispatch (`_SUBSTRATE_CLASSES`) selects by provision-id at boot time. Future Phase E MSAF real-wire follows the same pattern.

The choice to **reuse `WorkspaceBootError` rather than introduce a new `SessionExecutionError` typed exception** follows D48 §B.1 + D50 §B.1 precedent: category-vocabulary extension (over new exception class) when the recovery contract aligns. `sdk-init` failures share `WorkspaceBootError`'s abort-and-discard semantics with boot-time failures — the session never fully started; no observable state survives; caller's response (catch + decide retry/abort) is identical in shape. Introducing a new exception type here would force callers to catch two exception classes for what is structurally one recovery contract. Future Phase C+ runtime failures with distinct recovery semantics (e.g., mid-session SDK errors that DO leave observable state) may warrant a new typed exception; that's a future-entry call.

The choice to **wrap the async SDK in `asyncio.run` rather than introduce an async substrate model** honors D44 "No async substrate model" + D48 §D D-4 + D50 §D D-4. Async substrate is a Phase C+ cross-cutting decision affecting `Substrate.append_event` / `_dispatch_event_to_subscribers` / `handle_skill` / adapter `call()` simultaneously — too large to introduce in C1 alongside the SDK integration. The `asyncio.run` wrapper is a known-tradeoff: it spawns a fresh event loop per `start_session` / `stop_session` call (mild overhead, doesn't matter for the test-substrate scale; would matter for high-throughput production where async substrate becomes load-bearing). The tradeoff is named honestly + the path to fix-it-when-real (Phase C+ async substrate) is locked.

The choice to **inject a `_sdk_client_factory` for test mockability** (rather than monkey-patch the SDK module via `monkeypatch`) keeps the substrate self-contained + makes the injection point explicit. Tests inject a fake client class via instance assignment (`substrate._sdk_client_factory = _FakeClass`); the substrate resolves the factory at `start_session` entry; no global SDK-module monkeypatching is required. Per scope-cut C12 mockable-only discipline.
