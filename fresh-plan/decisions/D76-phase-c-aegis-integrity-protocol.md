## B. Triad applied per path

### B.1 — Event-chain integrity stamp + verify (NEW shape per D76; ACTIVATES D40 §B integrity-mechanism extension point)

| Triad element | Lock |
|---|---|
| **Detection** | Three classes of detection point: (1) **At stamp time** — `AEGISIntegrityProtocol.stamp(event, prev_hash)` computes SHA-256(JCS(event) || prev_hash) via `rfc8785.dumps()` + signs the hash bytes with the per-workspace Ed25519 private key. JCS canonicalization failure (non-canonicalizable event input) raises `IntegrityViolationError` with subcase=`canonicalization-error`. (2) **At verify time** — `AEGISIntegrityProtocol.verify(events, records, public_key)` walks events + records in parallel: (a) **missing-record** — records iterator exhausted before events (chain longer than records file); (b) **hash-mismatch** — recomputed `SHA-256(JCS(event) || prev_hash)` differs from `record.hash` (event content modified after stamp OR record file modified); (c) **signature-invalid** — `Ed25519PublicKey.verify(signature_bytes, expected_hash_bytes)` raises `InvalidSignature` (signature corrupted OR wrong public key). Verify stops at first violation per event (multiple sub-cases on the same event cascade). (3) **At boot time** — `boot.py` step 4.5+ invokes `.verify()` on the replayed chain; verify returns `list[ValidationFailure]`; non-empty list → bubble up. |
| **Surface** | `ValidationFailure(category='integrity-violation', path=<events.integrity.jsonl[sequence=N]>, value=<event-id>, reason=<structured-detail-string>)`. The `reason` field carries `subcase=<sub-case-name>` + `event-id=<offending-id>` + sub-case-specific structured detail (e.g., `expected-hash=<hex>` + `actual-hash=<hex>` for hash-mismatch). At boot time the list wraps as `WorkspaceBootError(failures=[...])` per D46 pattern + D45 standing triad. Direct callers of `IntegrityProtocol.verify` receive the list directly (typed surface). `IntegrityViolationError` is the typed exception class for stamp-time failures (canonicalization-error); boot.py + PersistenceLayer convert verify-time list-returns into the appropriate wrapper exception per call site. |
| **Recovery** | Caller catches `WorkspaceBootError` + inspects per-failure reason for sub-case + decides resolution per sub-case: **hash-mismatch** → chain content was tampered post-stamp; operator chooses to restore from prior backup, abort the workspace, or surface forensic detail for incident review (no automated repair — the chain is by definition compromised); **signature-invalid** → either signature corruption (bit-flip on disk; same restore-from-backup recovery) or wrong public key (operator verifies `integrity-public-key.pem` matches the workspace's stamping key); **missing-record** → integrity records file truncated; operator either restores from backup or starts a fresh workspace per the same recovery path as `persistence-corruption` (D70 §B.1); **canonicalization-error** → event content contains non-canonicalizable data (e.g., NaN floats, non-JSON-serializable types); upstream bug — operator surfaces via diagnostic API. Framework does NOT auto-repair; per D2 + D40 §B framing the integrity protocol is fail-closed: detection surfaces, operator decides recovery path. Per scope-cut C12: production key management (KMS / HSM) replaces local PEM at Phase D+; the recovery surface widens then with KMS-backed key-rotation + KMS-detected signature failure paths. |

- **D-5 — Rollback on integrity-record write failure**: Current behavior: if `events.jsonl` append succeeds but `events.integrity.jsonl` append (or the stamp() call itself) fails, the in-memory chain + on-disk events file are inconsistent with the on-disk integrity records file. `PersistenceCorruptionError` raises immediately surfacing the inconsistency, but the events.jsonl line is not retracted. Resolution requires a transaction abstraction that does not exist at Phase C scope.

Encrypted private-key storage (PEM with password-derived encryption; OR HSM-backed unwrap-on-use) deferred to Phase D+ KMS handoff.
