## B. Triad applied per path

### B.1 — Persistence file IO / parse / partial-line failure (NEW Phase C forward-bar)

| Triad element | Lock |
|---|---|
| **Detection** | `PersistenceCorruptionError` raised by `PersistenceLayer.load_chain` / `load_manifest_snapshot` / `save_event` / `save_manifest_snapshot` on: (a) directory create failure (permission denied); (b) `open()` failure (file vanished mid-read; permission denied; FS error); (c) `json.JSONDecodeError` on a line in events.jsonl (malformed object; partial-line at EOF from torn append); (d) parsed JSON line is not an object (e.g., scalar value); (e) parsed manifest snapshot is not an object. Error carries `path` + `reason` + optional `line_number` (1-indexed for events.jsonl line-level failures). Constructor-level type/serialization failures (`TypeError` from `json.dumps`) are also wrapped as `PersistenceCorruptionError` per the principle that the file's canonical state is now unreachable. |
| **Surface** | Boot.py wraps each `PersistenceCorruptionError` as `WorkspaceBootError(failures=[ValidationFailure(category="persistence-corruption", path=<error-path>, value=<line-number-or-None>, reason=<error-reason>)])` with the original `PersistenceCorruptionError` chained via Python's `from` clause (`__cause__`). Three call sites in boot.py (chain replay; manifest snapshot load; manifest snapshot save at end-of-boot). `WorkspaceBootError` reused (no new typed exception) per D48 §B.1 + D50 §B.1 + D69 §B.1 category-vocabulary-over-new-typed-exception precedent — `persistence-corruption` shares `WorkspaceBootError`'s abort-and-discard semantics with other boot failures. |
| **Recovery** | Caller catches `WorkspaceBootError`, inspects `failures[0].category == "persistence-corruption"`, decides whether to: (a) fix the underlying file state (e.g., truncate the torn line; restore from backup); (b) delete the persistence directory `<persistence-root>/<workspace-id>/` for a fresh-start boot (loses chain history; appropriate when corruption is irrecoverable); (c) abort and surface to operator. Substrate is NOT constructed when corruption surfaces during chain replay (boot raises before step 5 binding); chain integrity preserved per D10 (no partial chain leaks into the in-memory state). |

## D. What is NOT in this decision

The seven explicit DEFERS surfaced by the pre-lock probe (per probing.md Procedure 3 + D48 §D + D50 §D + D52 §D + D66 §D + D69 §D pattern):

- **D-1 — Snapshot caching strategy beyond full-replay-every-boot** (per D58 §D D-5): Phase C scope is one full chain replay per boot. Replay-cost framing is acceptable for Phase C reference-impl scale (per D68 §B.2 — JSONL chosen as the cheapest replay path; SQLite swap path named as the latency-bottleneck migration). When real-wire chain sizes produce unacceptable boot latency (e.g., >100ms replay time per D68 §B.2 alternative), a future entry adds snapshot caching with cache-invalidation strategy. NOT a closed contract — Phase C empirically discovers whether the bottleneck surfaces.

- **D-2 — In-memory rollback on persistence-write failure**: current behavior (substrate.py:297-305) propagates `PersistenceCorruptionError` from `save_event` to the caller WITHOUT rolling back the in-memory chain append at step 4. The in-memory chain remains source-of-truth; the caller sees the write failure as a runtime exception. Alternative (rollback the in-memory event_chain.append + state mutation on persistence failure) preserves "chain equals file" invariant but introduces atomicity complexity (two-phase commit semantics in `append_event`). Deferred to Phase C+ if file-truth-vs-memory-truth divergence surfaces as a real operational concern.

- **D-3 — Composition delta semantics between persisted state and new manifest**: when the replayed chain has actors AND the new manifest declares `composition.actors[]` differently (actor removed from manifest but present in chain; metadata diff for same actor-id; new actor in manifest not yet in chain), current behavior is "trust the chain" (manifest-actor seeding skips when `state.has_actor(aid)`). Full delta semantics (declare-but-not-in-chain → seed; in-chain-but-not-in-manifest → leave-alone OR flag; metadata diff → flag) deferred to Cref OR post-Cref clarification entry. The `state.has_actor` guard at `boot.py:659-668` is a minimal correctness fix preventing event-id collision on re-boot, not a complete composition-delta contract.

- **D-4 — Schema evolution across sessions**: events persisted under event.schema.json version N may fail schema validation when replayed against version N+1 (a slot becomes required; an enum value retired). Current behavior surfaces such failure as `persistence-corruption` (the file contents no longer match the framework's current schema). Schema-version field on each JSONL line OR a separate migrate-on-load mechanism deferred to future entry. Phase C bounded assumption: event.schema.json doesn't evolve mid-Phase-C; carry over to Cref retro-validation check.

- **D-5 — Cross-platform durability + fsync per append**: Phase C scope = POSIX append atomicity (Linux + macOS — events well under PIPE_BUF 4KB). Windows behavior (different filesystem semantics; lock-on-write conflicts) deferred to Phase D or later if pioneer-instance needs it. `fsync` per append for crash-durability is also deferred — current `flush+close` is sufficient for the Phase C reference-impl reliability bar; production-grade workspaces may need fsync per append OR a write-ahead-log discipline.

- **D-6 — Configurable persistence-root default**: current default is `./.fresh-plan-state/` (cwd-relative; matches Python convention; per D68 §B.2). Phase D pioneer-instance (PBS-Schulz) may need an OS-conventional default (e.g., `~/.local/share/fresh-plan/` per XDG-Base-Directory). Configurable per workspace via `composition.substrate-bindings[0].configuration.persistence-root` per D57 §B.1 opaque pass-through; default change deferred until Phase D surfaces concrete need.

- **D-7 — Multi-substrate persistence isolation + workspace-id uniqueness**: current contract is one persistence directory per workspace-id (`<persistence-root>/<workspace-id>/`). Two workspaces with the same `manifest.id` running concurrently against the same persistence-root would corrupt each other's events.jsonl. Workspace-id uniqueness is workspace-author responsibility per current contract; no framework-level lock or enforcement. Phase D federation work (per D26 + D7) may surface a need for content-hash-based persistence-key OR per-process lockfile; deferred.

Other items NOT in this decision:

- **No retroactive rewrite of D7, D10, D11, D12, D39, D40 §A, D54, D57, D58 entries** — append-only ledger discipline. D70 EXTENDS D10 + D39 + D40 §A by providing the persistence-layer contract those entries deferred ("persistence is implementation per D11"); EXTENDS D54 §D D-1 + D58 §C by ACTIVATING the previously-deferred boot integrations.
